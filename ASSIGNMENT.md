## Часть 1. Календарь встреч компании

Веб-приложение для планирования встреч внутри команды. Сотрудник добавляет встречу, указывает участников и время - система проверяет конфликты по всем участникам и блокирует бронирование, если кто-то занят.

**Демо:** https://latte-test.onrender.com/login

Данные для входа: `worker` / `worker123`

_Сервис на бесплатном хостинге - может потребоваться 30-60 секунд на первый запуск (холодный старт)._

**Репозиторий:** https://github.com/Huginn41/latte_test

### Стек и почему именно так

**FastAPI** - быстрый старт, автодокументация, встроенная поддержка Pydantic для валидации входных данных. Для такого проекта избыточный фреймворк вроде Django был бы лишним.

**Jinja2 + SSR** - серверный рендеринг без фронтенд-сборщиков. Шаблоны с наследованием (`base.html` -> `extends`) и переиспользуемые частицы через `include`. Vanilla JS только там, где нужен - отправка формы через `fetch` с инлайн-показом конфликта без перезагрузки страницы.

**PostgreSQL + psycopg2** - надёжное хранилище с поддержкой индексов по времени. `RealDictCursor` для удобного доступа к строкам как к словарям.

**Архитектура** - repository pattern: весь SQL только в `repository.py`, роутеры вызывают функции репозитория. Проверка конфликтов через SQL-перекрытие: `start_time < new_end AND end_time > new_start` по всем участникам встречи.

**Деплой** - Render (бесплатный web service) + Supabase (бесплатный PostgreSQL).

## Часть 2. Автоматизация процесса в fashion e-commerce

### Анализ рекламных кампаний

**Что и почему**

В fashion e-commerce рекламный бюджет распределён между несколькими площадками - Яндекс.Директ, VK Ads, маркетплейсы, иногда Meta. Каждая площадка даёт свою аналитику, и сводить это вручную в Excel раз в неделю - стандартная практика, которая плохо масштабируется. Аналитик тратит несколько часов на сбор цифр вместо выводов, а аномалии (резкий рост CPO, слив бюджета на нерабочий креатив) замечают постфактум.

Идея: ИИ-агент каждое утро собирает данные по всем кампаниям, сравнивает с предыдущим периодом и пишет короткий отчёт - что выросло, что упало, где стоит перераспределить бюджет. Для аномалий - алерт в мессенджер сразу, не раз в неделю.

**Что нужно**

Данные:
- API Яндекс.Директ и VK Ads для выгрузки метрик (показы, клики, расходы, заказы, CPO, ROAS) в разрезе кампания / объявление / день
- данные о продажах из CRM или 1С для расчёта реального ROAS, а не только платформенного
- исторические данные за 3-6 месяцев как базис для сравнения

Инструменты:
- cron-скрипт для ежедневной выгрузки данных в базу
- LLM (Claude или GPT-4o) для интерпретации - передаём таблицу с метриками и просим выделить отклонения и сформулировать гипотезы
- пороговые правила для алертов: CPO вырос больше чем на 30% за день - уведомление сразу, без ИИ
- Telegram-бот или простой дашборд для доставки отчёта команде

**Как проверить, что работает**

Первый месяц - агент пишет отчёты параллельно с ручной аналитикой. Сравниваем: совпадают ли выводы, какие аномалии агент поймал раньше человека, какие пропустил. Если агент находит те же проблемы, что аналитик, и делает это быстрее - переходим в боевой режим.

**Метрика через месяц**

Основная - время от возникновения аномалии до реакции команды (target: с 2-3 дней до нескольких часов). Вторичная - ROAS по портфелю кампаний: при том же бюджете должны расти, если мы быстрее перекладываем деньги из неработающего в рабочее. Дополнительно: сколько часов в неделю аналитик тратит на сборку отчётов (target: сократить с 4-6 до 1).

### 2. Tilda - VK Webhook

**Что это:** Сервис для получения заявок с сайта на Tilda в личные сообщения VK.

Задача возникла конкретная: нужно было принимать заявки в VK, потому что Telegram тогда активно блокировался, а нативного решения у Tilda не было. Написал за вечер - принимает POST от Tilda, форматирует заказ в читаемое сообщение и отправляет нескольким получателям через VK API.

**Репозиторий:** https://github.com/Huginn41/tilda-vk-webhook

**Роль:** Сделал полностью сам.

**Стек:** FastAPI, VK API, Docker, деплой на Railway.
